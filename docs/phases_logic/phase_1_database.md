# Phase 1: Database Migrations & Foundation

## الهدف والمغزى
هذه المرحلة هي حجر الأساس للمشروع بأكمله. تم فيها تصميم هيكل قاعدة البيانات (Schema) في PostgreSQL عبر Supabase وبناء الـ Migrations التي تنشئ الجداول، العلاقات، وصلاحيات الوصول (RLS).

## الملفات المرتبطة
- `supabase/migrations/001_create_profiles.sql` إلى `009_create_payments_and_featured.sql`
- `supabase/seed.sql` (لإدخال البيانات الأساسية مثل التصنيفات وباقات الاشتراك الافتراضية)

## اللوجيك (كيف تعمل)
- **Migrations**: بدلاً من إنشاء الجداول يدوياً، نكتب ملفات SQL. عندما نقوم بتشغيل المشروع، تُنفذ هذه الملفات بالترتيب. هذا يضمن أن بنية قاعدة البيانات متطابقة في كل بيئة (تطوير، إنتاج).
- **Row Level Security (RLS)**: ميزة قوية جداً في Supabase. بدلاً من كتابة كود في الـ Backend للتحقق من أن "المستخدم X يمكنه فقط رؤية بياناته"، يتم وضع هذه القاعدة مباشرة في قاعدة البيانات. هذا يمنع أي تسريب للبيانات حتى لو كان هناك خطأ في الكود.
- **Triggers**: استخدمنا الـ Triggers لأتمتة المهام. مثلاً، عندما يُسجل مستخدم جديد عبر Supabase Auth، يعمل Trigger لإنشاء ملف شخصي (Profile) له تلقائياً في جدول `profiles`. وتحديث `updated_at` تلقائياً عند أي تعديل.

## الـ User Flow (رحلة المستخدم في التطبيق)
هذه المرحلة لا يراها المستخدم مباشرة، ولكنها تدعم كل أفعاله:
- عندما يفتح التطبيق ويرى التصنيفات (Categories)، فهي قادمة من الـ `seed.sql`.
- عندما يحاول مستخدم تعديل بيانات مستخدم آخر، تمنعه قاعدة البيانات مباشرة (RLS).

## كيفية الصيانة والتعديل (Maintenance)
- **إضافة جدول جديد**: لا تقم بتعديل الملفات القديمة (من 001 إلى 009). بدلاً من ذلك، أنشئ ملف جديد (مثلاً `010_create_new_feature.sql`) واكتب فيه كود الـ CREATE TABLE.
- **تعديل جدول موجود**: أنشئ ملف جديد واكتب فيه `ALTER TABLE`.
- **تحديث الـ RLS**: إذا أردت تغيير صلاحيات الوصول، اكتب `ALTER POLICY` في migration جديد.
